# Madness Design Note

## Purpose

`狂気` まわりの実装方針を、実装担当がすぐ参照できる短い設計メモとしてまとめる。

## Core Direction

- `狂気` は `Power` を中心に扱う。
- 共通層依存は最小限に留める。
- 追加効果や特殊挙動は、できるだけカード個別の実装へ寄せる。

## What The Shared Layer May Handle

共通層で許容するのは、次の範囲までとする。

- `狂気` の消費タイミング管理
- Attack 解決時の `単体` / `ランダム` 対象を `全体` へ広げる処理
- 全体化の過程で起きる `target null` 回避

この範囲を超える追加効果やカード固有の解決順は、共通層へ寄せない。
`RandomReflectionCard` のように、`狂気` 時の「敵全体に 3 回」など対象変更以上の意味を持つ札は、カード個別で最終ターゲティングを持ってよい。

## What Should Stay Per Card

次のような要素はカード個別で扱う。

- 追加ダメージ
- 追加ドロー
- 特殊な対象条件
- 解決順に意味がある挙動
- 個別カードにだけ必要な例外処理

## Main Touchpoints

主な接点は次の 2 ファイルとする。

- `MadnessTargetTypePatch.cs`
- `MadnessPower.cs`

共通層の変更を検討する場合は、まずこの 2 点への影響から確認する。

## Review Rule For Future Cards

- 将来の `狂気` 関連カードは、個別レビュー前提で扱う。
- 新カード追加時は、「共通層へ足すべきか」「カード個別で閉じるべきか」を毎回確認する。
- 共通層へ処理を追加するのは、複数カードで再利用され、かつ副作用範囲を説明できる場合に限る。

## Next Smallest Slice

- 次に切る最小単位は、`狂気` の対象変更を `DokaanCard` と `RandomReflectionCard` の実挙動に限定して整理する小修正とする。
- この段階では共通層の一般化を広げず、2 枚のカードで必要な最終ターゲティングと解決差分を見極める。
- 検証観点は `狂気なし` / `狂気あり` の対象数差をこの 2 枚で見ることに絞る。
